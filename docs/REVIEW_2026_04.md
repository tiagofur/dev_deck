# DevDeck — Review técnico y de producto (abril 2026)

> Autor: revisión externa · Fecha: 2026-04-08 · Estado: input para planning de Ola 4.5 y Ola 5
>
> Este documento captura un análisis honesto del repo en el cierre de Ola 4, con foco en qué hay que arreglar **antes** de avanzar a Ola 5, y qué falta para que DevDeck pase de "buen proyecto de ingeniería" a "producto memorable".

---

## TL;DR

- **Visión y ejecución por olas: excelentes.** Posicionamiento clarísimo, roadmap disciplinado, stack sobrio.
- **Red de seguridad: inexistente.** Cero tests en backend y frontends. Esto es bloqueante para Ola 5.
- **Captura con fricción: el mayor riesgo de producto.** Sin extensión de browser / CLI / paste inteligente, DevDeck compite con `Cmd+D`.
- **Decisiones pendientes críticas:** modelo polimórfico de `items`, estrategia de sync (LWW vs CRDT), límites de costo de IA.
- **Higiene de repo:** binario `api.exe` commiteado, `ROADMAP.md` duplicado, sin screenshots en README, sin `CONTRIBUTING/SECURITY/SELF_HOSTING`.

Se propone una **Ola 4.5 — Hardening & Capture** antes de Ola 5 (ver `ROADMAP.md`).

## 1. Lo que está muy bien

### 1.1 Visión clara y diferenciada
"Memoria externa dev asistida por IA" es un posicionamiento entendible en 5 segundos. El análisis competitivo (GitHub Stars, Raindrop, Notion, Pieces, Raycast) explicita por qué ninguno cubre el caso del dev que quiere *redescubrir* lo que ya guardó. Pocos proyectos indie tienen este nivel de claridad estratégica.

### 1.2 Ejecución disciplinada
- Olas 1–4 completas con fases bien atomizadas.
- Migrations numeradas, endpoints documentados, dominios separados (`internal/domain/*`).
- El ROADMAP lee como un changelog real, no como wishful thinking.

### 1.3 Stack coherente
Go + Chi + pgx + Postgres en back, Electron+React y Vue en clientes, Caddy+Docker Compose para deploy. Nada caprichoso, todo production-ready, binarios chicos (distroless ~5 MB).

### 1.4 Detalles de producto que la mayoría ignora
- Mascota Snarkel con 5 moods y copy rioplatense.
- Discovery mode con swipe (tinder-like) y acciones por dirección.
- Atajos globales a nivel OS en Electron.
- `safeStorage` con keychain del sistema para tokens.
- Auto-refresh JWT con lock de concurrencia.
- Neo-brutalist consistente en ambos clientes.
- Confirm/toast singletons, PageTransition, ShortcutsModal.

Esto es **pulido real**, no maquillaje.

## 2. Problemas críticos (bloqueantes para Ola 5)

### 2.1 Cero tests — Riesgo: ALTO
Búsqueda `*_test.go` / `*.test.ts` / `*.spec.ts` fuera de `node_modules`: **0 archivos**. Con 4 olas ya hechas y el modelo de datos a punto de mutar radicalmente en Ola 5 (`repos → items` polimórfico), arrancar sin red de seguridad es la receta para romper cosas en silencio.

**Mínimo antes de Ola 5:**
- **Backend:** tests de handlers con `httptest` + Postgres dockerizado vía `testcontainers-go`. Cubrir al menos: `repos`, `commands`, `cheatsheets`, `auth`, `search`. Target: ≥ 60% de cobertura en `internal/http/handlers` y `internal/store`.
- **Frontends:** Playwright E2E con 5 flujos críticos:
  1. Login (OAuth stub).
  2. Add repo → aparece en grid.
  3. Ver detail → editar notas → guardar → recargar.
  4. Búsqueda global (Cmd+K) encuentra repo y cheatsheet.
  5. Discovery mode → swipe archive.
- **CI:** un workflow de GitHub Actions que corra `go test ./...` + `pnpm test` en cada PR. Sin CI verde, no se mergea.

Ver `docs/TESTING_STRATEGY.md` para el plan detallado.

### 2.2 `api.exe` commiteado al repo — Riesgo: MEDIO (higiene)
Hay un binario Windows de 5+ MB en `backend/api.exe`. Esto pudre el repo, dispara falsos positivos de antivirus, y ensucia `git log`. Acción: `git rm api.exe`, agregar `*.exe` + `api` + `api.bin` al `.gitignore` del backend, y escribir el build en `backend/bin/`.

### 2.3 `ROADMAP.md` duplicado — Riesgo: MEDIO (confusión)
El archivo tiene **dos versiones** de Ola 5 y Ola 6 (aproximadamente líneas 200–250 y 314–440). Fase 17 aparece dos veces con descripciones distintas. Esto confunde a cualquiera que llegue nuevo, incluido el propio autor en 3 meses. Acción: reconciliar en una sola versión canónica (ver sección 6 más abajo y `ROADMAP.md` actualizado).

### 2.4 Decisión del modelo `items` no formalizada — Riesgo: ALTO
Ola 5 propone migrar `repos` → `items` polimórfico con `item_type` + JSONB `meta`. No hay ADR. Sin decisión escrita y revisada, esta migración se va a hacer "a ojo" y va a arrastrar deuda por el resto del proyecto.

Ver `docs/adr/0001-items-polymorphism.md` para el análisis de alternativas.

## 3. Problemas importantes (no bloqueantes, pero caros si se ignoran)

### 3.1 Sync offline-first con last-write-wins es una trampa
"LWW por campo usando `updated_at`" suena simple pero tiene dos fallos clásicos:
1. Relojes de cliente desincronizados → órdenes falsos.
2. Dos devices editando `notes` (texto libre) → el último pisa al anterior en silencio. Pérdida de trabajo real.

**Recomendación:**
- Timestamps **del servidor**, no del cliente, para ordenar operaciones.
- `client_id + operation_id` para idempotencia (ya está en el plan — bien).
- Para el campo `notes` específicamente, evaluar **CRDT liviano** (Yjs o Automerge). Para el resto de los campos, LWW con server timestamp es suficiente.
- Sync log auditable (ya planeado en Fase 22).

### 3.2 Búsqueda híbrida: evitar suma ponderada
`pg_trgm` devuelve similarity ∈ [0,1], `pgvector` devuelve distancia coseno ∈ [0,2]. Combinarlos con un peso lineal (`0.4*trgm + 0.6*cosine`) da resultados inestables porque las escalas no son compatibles.

**Usar Reciprocal Rank Fusion (RRF):** rankeá por cada métrica por separado, y combiná con `Σ 1/(k + rank)` (k ≈ 60). Son 5 líneas de SQL y el resultado es dramáticamente mejor que cualquier ponderación lineal. Referencia: el paper de Cormack et al. 2009.

### 3.3 Costos de IA sin límites
Ola 5/6 menciona rate limiting recién en Fase 24. Es tarde. Un user con 5000 items puede quemar créditos de OpenAI en una tarde si el pipeline de enrich se dispara en bulk.

**Recomendación, desde la Fase 18:**
- Hard cap por usuario: `embeddings/day`, `summaries/day`, `ask/day`.
- Contador visible en Settings: "Usaste 312 / 1000 embeddings este mes".
- **Ollama como default**, OpenAI como opt-in. Invertir el framing actual.
- Circuit breaker: si el spend mensual supera $X, desactivar IA y notificar.
- Cachear embeddings por hash del contenido (si no cambió, no re-embeddear).

### 3.4 Seguridad: revisar antes de multi-user
- **SSRF en el Open Graph scraper:** un user malicioso puede meter `http://169.254.169.254/...` (AWS metadata) o `http://10.0.0.1/` y leer red interna. Whitelist de esquemas (`http`/`https`), resolución DNS previa, bloqueo de rangos RFC1918, RFC6598, RFC3927, loopback, link-local.
- **Rate limit global en `/api`:** Chi tiene `httprate` — 5 líneas.
- **CSP en renderer de Electron:** verificar `contextIsolation: true`, `nodeIntegration: false`, `sandbox: true` donde sea posible.
- **`ALLOWED_GITHUB_LOGINS` no escala:** cuando pases a multi-user real (Ola 7), necesitás flow de waitlist/invites, no env vars.

### 3.5 Observability ausente
No hay métricas, tracing ni logs estructurados. Aunque seas self-hosted, meté desde ya:
- `slog` estructurado (Go 1.23 lo tiene built-in).
- Endpoint `/metrics` Prometheus con latencias por handler + contadores de errores.
- Tracing opcional con OpenTelemetry (solo si hay endpoint configurado, sino noop).
- Logs de IA: costo por request, latencia, provider, user.

Te va a salvar cuando algo ande lento y no sepas si es la IA, pgvector, el enricher o GitHub rate-limitándote.

## 4. Lo que falta para que sea "increíble"

Más allá del roadmap existente, esto es lo que sube a DevDeck de "linda app dev" a "herramienta que no podés dejar de usar".

### 4.1 Captura sin fricción es EL todo
Si agregar algo toma más de 3 segundos, nadie lo va a usar. Necesitás:

**(a) Extensión de browser (Chrome + Firefox).** Atajo que captura URL activa + selección de texto + meta tags + manda a DevDeck vía API. Sin esto, competís con `Cmd+D` y perdés.

**(b) CLI `devdeck`.** Para devs que viven en terminal:
```bash
devdeck add https://github.com/...
devdeck add --type=cli "ripgrep"
devdeck search "debugging go"
devdeck run <alias>       # ejecuta un comando guardado
echo "foo" | devdeck capture --type=snippet
```
Esto habilita integraciones con git hooks, aliases de shell, `brew install` wrappers, etc.

**(c) Paste inteligente.** Pegás cualquier cosa en la home y DevDeck detecta el tipo:
- URL GitHub → repo.
- URL cualquiera → article/tool.
- Bloque de código → snippet.
- Línea que empieza con `$` → command.
- JSON → prompt/agent config.

**(d) Share target en PWA Vue.** Cuando el web client sea PWA, registrar share target handler. Así en móvil "Share → DevDeck" funciona nativo sin app nativa.

Ver `docs/CAPTURE.md` para la spec detallada.

### 4.2 "¿Por qué lo guardé?" obligatorio pero invisible
Ese campo es tu moat real — GitHub Stars nunca lo va a tener. Pero un textarea vacío no lo llena nadie. Hacé que la IA **proponga 3 razones** al guardar:
- "para agents en terminal"
- "para debugging Go"
- "atajo productividad Mac"

El user elige con **un click**, no escribe un párrafo. Si ninguna encaja, hay opción "otro → escribir". Esto convierte un campo opcional en default rellenado.

### 4.3 Integración con el flujo real del dev
- **Git hook** que detecta cuando clonás un repo localmente y pregunta "¿guardarlo en DevDeck?".
- **Watcher de `~/code`** opcional.
- Leer `history | grep "brew install\|npm install -g\|cargo install"` y proponer CLIs para capturar.
- `devdeck exec <alias>` con **variables**: `devdeck exec deploy --env=staging`.

### 4.4 Runbooks ejecutables, no solo checklist
Si un runbook es "instalá X, corré Y, configurá Z", que el botón **Run** los ejecute secuencialmente en un terminal embebido, con confirmación por paso. Ahí pasa de "doc bonita" a "automatización". **Security crítica:** sandboxear, whitelist de comandos, nunca ejecutar comandos del server, solo locales del cliente.

### 4.5 Vault score + review semanal
Usá la mascota. Notificación los domingos: *"esta semana guardaste 12 items. 3 están sin abrir desde febrero. ¿Los revisamos?"*. Esto convierte DevDeck en **hábito**, no en cementerio de bookmarks. Es lo que Raindrop/Pocket nunca supieron hacer.

### 4.6 Import masivo el día 1
Sin importadores, un user nuevo entra a una app vacía y se va.
- GitHub Stars (el más obvio).
- Raindrop export (JSON).
- Pocket export (HTML).
- `~/.bash_history` filtrado.
- Browser bookmarks (Chrome/Firefox JSON).
- `gh repo list` del usuario.

### 4.7 Docs que faltan para un proyecto open-source serio
- `CONTRIBUTING.md` — cómo correr local, estilo de commits, cómo proponer PRs.
- `SECURITY.md` — cómo reportar vulnerabilidades.
- `docs/SELF_HOSTING.md` — guía paso a paso.
- Screenshots / GIFs en el README. Para una app tan visual, no tenerlos es pecado capital marketing-wise.
- Un post estilo "Cómo construí DevDeck" para HN/Lobsters cuando Ola 6 esté lista.

## 6. Resumen ejecutivo para el owner

DevDeck está en un **punto raro y valioso**: la arquitectura y la visión están *sobre-desarrolladas* respecto al testing y a la captura. La ingeniería es fuerte, el posicionamiento es claro, el pulido de UX es real.

**Lo que falta no es más features** — es:
1. **Red de seguridad** (tests + CI) para poder iterar rápido sin romper.
2. **Captura sin fricción** (extensión + CLI + paste) porque sin eso, el producto no se usa.
3. **Decisiones formalizadas** (ADRs) para no pagar deuda técnica en Ola 5.
4. **Higiene de repo** (dedupe docs, sacar binarios, screenshots) para que se vea serio desde afuera.

Con 4 semanas de Ola 4.5 bien ejecutadas, Ola 5 y 6 vuelan. Sin ellas, cada ola futura cuesta el doble.

**Ahora mismo DevDeck es un gran proyecto de ingeniería. Lo que le falta para ser un gran producto está casi todo del lado del onboarding, la captura y el hábito.**
